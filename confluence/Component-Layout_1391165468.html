Overview

The following will act as a guide for how we expect components to be structured per page. In the context of this project a “Page” refers to most react router endpoints or the space between the header and footer components. Each page tries to match the Figma drawing which helped dictate component structure

Main.tsx/App.tsx

Main is the starting point for the application where we create the browser router and assign its associated provider to the rest of the application.

Internal Components

Main contains the following components and subcomponents:

  • App.tsx

    • Context Providers:

      • ConfirmationContext.tsx

      • UserContext.tsx

      • SearchContext.tsx

      • SasTokenContext.tsx

      • ProgressContext.tsx

      • AlertContext.tsx

    • Layout Components:

      • Header.tsx

      • Footer.tsx

      • Outlet

Outlet is a placeholder component that outputs the child components of the parent (the parent in this case is app.tsx

Main provides the react router to app.tsx (making app the parent) which in turn gives access to all child components of app.tsx

App.tsx is the top level of the router which all other component will render inside of.

UI Libraries used:

  • Dropdown: There are two dropdowns on the asset display page. The first being under the h2 title and one with the pagination. It is through a library called Radix . It is a highly customisable library with a sufficient CSS properties that can be seen on the link above.

  • ChevronDownIcon, ChevronRightIcon are both a part of the icon library called heroicons. These can installed separately or one can copy the svg files and use it directly. I however decided to install to make the code look cleaner.

  • SideNav or Sidebar: The Sidebar is using DaisyUI's Drawer component. There are some quirks I had to use for making the colors referenced in tailwind.config.js work with colours in the sidebar and other things that might be useful to know.

    • If you want to change the colors of the checkbox colours there are properties called --chkfg and --chkbg for changing the color of the tick mark or the background colour respectively.

      • These can be referenced directly in the tailwind however, it is not possible to use the custom colour codes in tailwind.config.js like general-10 or yellow-80 with these properties.

      • To solve this I created a separate colors.ts file, referenced it in the tailwind.config.js as colors: colors under theme. Next, I imported the colors.ts file in the index.css and in the :root I can use the same color codes with -- before them like --general-10: theme("colors.general.10");. Theme() fetches the color and assigns it to the custom css variable created. This custom variable is what I can use in the AssetDisplay.css file like 

        .checkbox {
  --chkbg: var(--general-50);
}

        . checkbox assigns the css to all the checkboxes which is what we are targetting. --chkbg is the property of the checkbox from DaisyUI. and var() grabs the --general-50 custom css var we created in the index css. Annoying workaround but it works

    • Many components in DaisyUI have default padding and default rounding which has to be changed by explicitly mentioning p-0 or rounding-none in the tailwind classes.

AdminPage.tsx

The admin page will display all functionality that is locked behind “Super-Admin” privilege, these functionalities include:

  • Changing user roles / deleting users

  • Managing projects

    • Create

    • Delete

  • Upload Asset

    • Create

Internal Components

AdminPage.tsx will contain the following components and elements

  • Button

  • UserManagement.tsx

  • ProjectManagement.tsx

    • AddProject.tsx

  • UploadAsset.tsx

  • Pagination.tsx (Dynamic render on only Project and User Management)

  • Dropdown

The page will have an internal container which will deploy sub-components related to each tab (User Management, Project Management, and Uploading Assets).

It might be the case that rendering the drop down and pagination inside of the sub-components of AdminPage is better. Proceed at your own discretion

UserManagement.tsx

The User management component is a tab of the admin page that will allow the admin user to change the role of each registered user and be able to delete the user.

ProjectManagement.tsx

The project management component is a tab of the admin page that will allow the user to delete or add projects to the database. It is the only way for the client to directly manipulate the database for information related to projects.

The modal component exists as a separate component inside of the page sub-component. See the section on Modal.tsx for more information

AddProject.tsx

This is a child component to be given to the modal component that will appear as a pop-up on top of the admin page. All sub elements should exist inside of a form that is submitted using the create button.

The Specific part to mention here is the FileUploader.tsx which should appear as a separate JS library component (similar to carousel) see https://www.npmjs.com/package/react-drag-drop-files for an idea of a node library that can accomplish this functionality.

UploadAsset.tsx

The upload asset component is a tab of the admin page that will allow the user to add assets to the blob storage and database. It is the only way for the client to directly manipulate the database for information related to assets and the only way for a user to upload a new asset.

All sub elements should exist inside of a form that is submitted using the create button.

The Specific part to mention here is the FileUploader.tsx which should appear as a separate JS library component (similar to carousel) see https://www.npmjs.com/package/react-drag-drop-files for an idea of a node library that can accomplish this functionality.

Missing from the drawing is Preview Images with a file uploader and Project with a dropdown to select a project to assign the asset to.

Modal.tsx

The modal component is a container that uses portals to render to the modal div directly inside of the index.html

https://legacy.reactjs.org/docs/portals.html See portals for more information

Modal component should be present in each component that will be popping up the modal. For instance inside of AssetDisplayPage.tsx we have AssetCards will call a function that will open the dialog (via refs) and set the internal state of the modal to render. Inside of the modal component are conditional render statements looking at the isOpen prop to determine if it should render the child elements of the modal. The state that manages the rendering the child elements of each modal should be in the parent relative to where the modal is (so in this case in the AssetDisplayPage.tsx). The Modal component expects child elements (in the example of AssetDisplayPage it would be the AssetDetails Component) Which it will render as a subcomponent of itself, this making it modular to whatever you wish to put inside.

DisplayCard.tsx

The display card component is a container for images and text related to each asset or project. It is a simple card that will display information and ideally pop-up a more detailed modal or page related to each asset or project.

Pagination.tsx

The pagination component is a React component for pagination that displays a series of page numbers and allows the user to jump to any page by clicking on the page number. The component also displays arrow icons for navigating between pages.

NotFoundPage.tsx & UnauthorizedPage.tsx

The NotFoundPage.tsx & UnauthorizedPage.tsx page displays an animated 3D component intended for rendering 404 or 403 error pages using Three.js and React Three Fiber. It displays a rotating 3D error code (404 by default, and 403 for unauthorized error).
