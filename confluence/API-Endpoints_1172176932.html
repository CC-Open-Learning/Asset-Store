API Calls

This section will contain information regarding the calls the API currently accepts as well as information related to how to make calls and some of the back-end routing

Server URL

The API for the server can be called by entering the following base URL

https://assetstore.vconestoga.com/api

Asset Routing

Assets can be fetched by appending “asset” to the end of the URL as shown below.

https://assetstore.vconestoga.com/api/asset

Route

Functionality

/download

Serves the download URL for the asset

/asset-sas-token

Serves the SAS token for the assets

/

.delete + root path deletes the asset by ID

/

.post + root path uploads an asset

Ex. GET Request “/download”

https://assetstore.vconestoga.com/api/asset/download?id=666b882572de3e184b03eccf

File Download Endpoint Flow

  1. User Request:

    • User sends a GET request to /download with the id of the asset in the query.

  2. Authentication Check:

    • The checkAuthentication middleware verifies the user's authentication.

  3. Extract File ID:

    • The endpoint retrieves the id from the query string.

  4. Find Asset in MongoDB:

    • Searches for the asset metadata using assetModel.findById(id).

    • If not found, returns a 404 error with a "Asset not found" message.

  5. Generate SAS Token:

    • If the asset exists, generates a temporary SAS token for secure access to the file.

  6. Create File URL:

    • Combines the Blob Storage URI, container name, file name, and SAS token to create the full file URL.

  7. Check Blob Existence:

    • Verifies the file exists in Blob Storage using a BlobClient.

    • If it doesn’t exist, returns a 404 error.

  8. Log Download:

    • Logs the username and file name of the download request.

  9. Respond with File URL:

    • Sends the file URL with a 200 status if the file exists.

  10. Error Handling:

    • Returns a 500 error for any unexpected issues during the process.

The fileUrl is the URL from the download endpoint in the server (/download).

Download route code:
router.get("/download", checkAuthentication, async (req, res) => {
  try {
    const { id } = req.query;
    const asset = await assetModel.findById(id);
    const minuteDuration = 2; // Change duration according
    if (asset !== null) {
      // Generate SAS token for the asset file
      const sasToken = await createContainerSasToken("assets", minuteDuration);

      // Construct the full URL
      const fileUrl = `${process.env.BLOBSTORAGE_URI}assets/${asset.fileName}?${sasToken}`;

      // Create a BlobClient using the full blob URL with SAS token
      const blobClient = new BlobClient(fileUrl);

      // Check if the blob exists
      if (await blobClient.exists()) {
        //Log user who requested the download
        logger.info(
          `User ${req.user.username} requested download for "${asset.fileName}"`
        );

        // Send the full URL as the response
        return res.status(200).send({ url: fileUrl });
      }
      return res.status(404).send({ message: "Asset not found" });
    }
  } catch (e) {
    logger.error(e.message);
    return res.status(500).send({ message: e.message });
  }
});

The URL returned to the client will be processed in the DownloadAssetButton.tsx where the URL is used like this to download the asset:

Frontend download code
const link = document.createElement("a");

link.setAttribute("download", fileName); // Set the desired file name for the download
link.setAttribute("href", fileUrl); // Use the provided file URL directly

// Append the link to the document and trigger a click

Delete Request

Full route:

https://assetstore.vconestoga.com/api/asset/?id=674cd4dd67044c6225186ebbz

Delete Asset Flow

  1. User Request:

    • User sends a DELETE request to / with the asset id in the query.

  2. Authentication and Authorization:

    • The checkAuthentication middleware validates that the user is authenticated.

    • The checkIsAdmin middleware ensures the user has admin privileges.

  3. Extract Asset ID:

    • The endpoint retrieves the id from the query string.

    • If id is not provided, responds with a 404 error: "Id not found."

  4. Delete Asset Logic:

    • Calls the deleteAsset(id) function to:

      • Remove the asset metadata from MongoDB.

      • Delete associated files (e.g., main files, previews) from Azure Blob Storage.

  5. Success Response:

    • If deletion is successful, returns a 200 status with the message: "Asset Deleted."

  6. Error Handling:

    • If an error occurs during deletion or file handling, logs the error message and responds with a 500 status.

Delete route code
router.delete("/", checkAuthentication, checkIsAdmin, async (req, res) => {
  try {
    const { id } = req.query;
    if (id) {
      await deleteAsset(id);
    } else {
      return res.status(404).json({ success: false, message: "Id not found" });
    }

    return res.status(200).json({ success: true, message: "Asset Deleted." });

    //delete MongoDB metadata
  } catch (e) {
    logger.error(e.message);
    return res.status(500).send({ success: false, message: e.message });
  }
});

Delete Request Utilities

delete.js

The main function for deleting. Deletes the file, preview images, and metadata from the database.

deleteAsset()
const deleteAsset = async id => {
  const asset = await assetModel.findById(id);

  if (asset) {
    const {
      _id: assetId,
      production,
      fileName: assetFileName,
      previews: previewImages,
      tags
    } = asset;

    try {
      // Attempt to delete blobs first
      if (production && production.glb) {
        await deleteAssetBlob(assetContainerName, production.glb);
      }
      await deleteAssetBlob(assetContainerName, assetFileName);
      await deletePreviewImages(previewContainerName, previewImages);

      // Proceed to delete database records only if all blob deletions succeed
      await assetModel.deleteOne({ _id: assetId });
      await deleteTags(tags);
    } catch (error) {
      console.error(`Error during asset deletion: ${error.message}`);
      logger.error(`Error during asset deletion: ${error.message}`);
    }
  }
};

Helper function to delete a blob from a specified container

deleteAssetBlob()
async function deleteAssetBlob(containerName, blobName) {
  const containerClient = blobServiceClient.getContainerClient(containerName);
  const blobClient = containerClient.getBlobClient(blobName);
  try {
    await blobClient.deleteIfExists({ deleteSnapshots: "include" });
  } catch (error) {
    console.error(`Error deleting blob ${blobName}: ${error.message}`);
    logger.error(`Error deleting blob ${blobName}: ${error.message}`);
  }
}

Main function for deleting preview images

deletePreviewImages()
//Remove preview image files from Preview-Images folder in storage
const deletePreviewImages = async (containerName, previewImages) => {
  try {
    // Ensure previewImages is an array
    const imagesToDelete = Array.isArray(previewImages)
      ? previewImages
      : [previewImages];

    const deletePromises = imagesToDelete.map(async previewUrl => {
      // Extract the blob name from the URL
      const blobName = previewUrl.split("/").pop();
      // Decode the blob name to handle any encoded spaces
      const decodedBlobName = decodeURIComponent(blobName);

      // Create a BlockBlobClient and delete the blob
      const containerClient =
        blobServiceClient.getContainerClient(containerName);
      const blobClient = containerClient.getBlobClient(decodedBlobName);

      await blobClient.delete(); // Delete the blob

      logger.info(
        `Deleted blob: ${decodedBlobName} from container: ${containerName}`
      );
      return true; // Indicate successful deletion
    });

    // Wait for all delete operations to complete
    const results = await Promise.all(deletePromises);
    return results; // Return results of delete operations (true/false)
  } catch (error) {
    logger.error("Error deleting preview images:", error);
    return false; // Indicate failure if an error occurs
  }
};

Search Routing

Any Asset in the database can be fetched by appending “search” to the end of the URL as shown below.

https://assetstore.vconestoga.com/api/search

Routes

Search has a series of sub-routes used to search assets, projects, tags, and asset categories. They are used by appending the name of the route to the end of the URL as shown below

https://assetstore.vconestoga.com/api/search/<route>

Route

Functionality

/

Search assets based on specified keywords: names, tags, categories, and projects.

/asset

Search by any known property (case insensitive & partial match)

Select for MongoDB filtering

/project

Search by project name (case insensitive & partial match)

/category

Search by category name (case insensitive & partial match)

/tags

Search by project name (case insensitive & partial match)

Most searches should be handled via the “/” route especially in regards to the bottom 3 routes as well as asset searches by NAME. All other more specific searches regards specific properties of a given asset (ex. model.triCount, fileSize, etc) needs to be sent via the “/asset“ route

Search by Asset ID

To search for an asset by ID specifically use the following route

https://assetstore.vconestoga.com/api/search/asset?_id=<id>

In the following example the ID is appended to the URL in order to be attached to the parameter part of the request body. This is different than all the above requests which use query

https://assetstore.vconestoga.com/api/search/asset?_id=66a284d5d87f9399c8855845

Search assets based on keywords

To search by key words the URL must have ?keywords appended to the end of the search route

To search for assets through keywords use the following route:

https://assetstore.vconestoga.com/api/search?keywords=[keyword1],[keyword2],[keyword3],...

The keywords are split by commas, spaces will effect search results.

Currently the search route (using keywords) only works by name, any more specific searches regarding assets (such as numbers) need to be made via the search/asset route above

A search query is constructed for the Asset collection using the following criteria:

  • name: Matches any asset name that contains any of the keywords (case-insensitive partial match).

  • tags: Matches any asset tag that contains any of the keywords (case-insensitive whole match).

  • categories: Matches any asset whose categories include any of the keywords (case-insensitive whole match).

  • projects: Matches any asset whose projects include any of the keywords (case-insensitive whole match).

Search Using Inequality Operators

When searching using inequality operators such as (less than, greater than…etc) the API is expecting each operator as an object part of the property you wish to search.

The API currently supports the follow searches:

Inequality

Code equivalent

Less than or equal to

lte

Greater than or equal to

gte

Example of URL

https://assetstore.vconestoga.com/api/search/asset?model[triCount][gte]=2&model[triCount][lte]=10000&fileSize[gte]=8000

In the above example the API interprets the search parameter you wish to use as an embedded object inside of the property you are searching for. For the property model.triCount, the gt property is being embedded inside as follows:

Example Query Object
{
  "model": {
    "triCount": {
      "gt": 2
      "lt": 10000
    }
  }
}

Sending a request such as this will cause values greater than 2 and less than 10,0000 to be returned

For a single search like with fileSize the query object would only contain a single object fileSize[gt]=8000

params: {

}

AXIOS Request

Below is an example of what the AXIOS get request should look like in order to get a response from the API using select:

Pure Axios GET Example
const axiosCall = async () => {
    await axios({
      method: "get",
      url: "https://assetstore.vconestoga.com/api/search/asset",
      headers: {
        "Content-Type": "application/json"
      },
      params: { 
        select: ["name"], 
        name: ["Air", "bag"] 
      }
    })
      .then(res => res.data)
      .catch(err => {
        console.log(err.message);
        return false;
      });
  };

Which produces the following URL:

https://assetstore.vconestoga.com/api/search/asset?name=Airtank&select=model.triCount name

The request can also be made by layering in the parameters as string literals in the URL of the fetch request. From the Asset Store client requests are made using Axios, therefore to keep things consistent it has been used in this example

The call as shown above shows how to include filter parameters (via select) and also regular search parameters for properties such as name

AXIOS Service Request

Below is an example of what the AXIOS get request should look like when using the Axios Service:

Service Axios Get Example
const byKeyword = async ({ keyword }: byKeywordParams): Promise<Asset[]> => {
  const queryParams: AxiosRequestConfig = {
    params: {
      keyword: keyword ? keyword.split(",").map(k => k.trim()) : [],
      select: "_id name updatedAt"
    },

    /**
     * Serializes the query parameters for the request.
     * @param params The query parameters to serialize.
     */
    paramsSerializer: params => {
      // Use URLSearchParams to serialize the params object
      const searchParams = new URLSearchParams();
      if (Array.isArray(params.keyword)) {
        searchParams.append(
          "keywords",
          params.keyword.join(",").replace(/ /g, "_").toLowerCase()
        );
      }
      if (Array.isArray(params.select)) {
        searchParams.append("select", params.select.join(","));
      }
      return searchParams.toString();
    }
  };
  // Use AxiosService to make the GET request

  return await AxiosService.GET<Asset[]>(baseRoute, queryParams);
};
