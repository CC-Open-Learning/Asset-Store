Functionality

AxiosService is a centralized service layer designed to manage all HTTP requests within the application. Using axios as its core, AxiosService streamlines API interactions with a modular structure, ensuring consistent request handling and error management across the project.

Each method in AxiosService corresponds to a specific HTTP operation, making it easy to interact with different API endpoints:

  • GET (endpoint, config?): Retrieves data from a specified API endpoint.

  • POST (endpoint, data, config?): Submits data to create a new resource.

  • PUT (endpoint, data, config?): Updates an existing resource with the provided data.

  • DELETE (endpoint, config?): Deletes the resource at the specified endpoint.

Parameters

Each method accepts the following parameters to simplify API interactions:

  • endpoint: The specific API path after /api (e.g., /users).

  • config (optional): An optional AxiosRequestConfig object, which allows customization of the request. It can include settings such as query parameters, custom headers, timeout, authentication, and more. This parameter gives flexibility to adjust requests based on specific needs.

  • data (only for POST and PUT): The JSON object containing the data to send in the request body, typically organized as key-value pairs.

Each method accepts an endpoint (the specific API path following the base URL) and an optional config for custom headers, query parameters, or other settings. The POST and PUT methods additionally take a data parameter to send a payload within the request body. If an error occurs during any request, it’s caught and processed through a shared error handler, handleAxiosError, which standardizes error management across all methods.

Return Value

Each method returns a Promise that resolves to:

  • AxiosResponse<T>: The server’s response, containing the requested or modified data. (See )

Error Handling

If an error occurs during any request, it is processed by the handleAxiosError function. This standardized error handler ensures that errors are managed consistently across all HTTP methods, improving debugging and logging.

Creating a new service with AxiosService

Getting Started

1. Create the Service Folder:

  • Navigate to the services directory in your project.

  • Create a new folder to organize your service files based on the purpose of the service. The folder name should reflect the functionality or category of the service.

  • Example: If your service interacts with the /search endpoint, create a folder named search.

2. Create the Service File:

Next we need to create the service file located in the folder created in the previous step. The service should be named by the category or theme of the service followed by the keyword "Service" . For example, for the folder search would contain the service SearchService.

3. Building the service:

Each function within the service will use AxiosService to make HTTP requests. Follow these conventions for creating modular, consistent, and type-safe functions.

Function Structure:

  • Define Parameters: Destructure parameters directly in the function arguments.

  • Return Type: Always define the return type as a Promise that resolves to either the expected data type or an error.

Templates

Types

In these templates resource is used as an example. You should be retrieving your types from the types.ts from the root of the client-portual. If your function will not return one of these types, you should instead create an interface based entirely on response expected from the API. For example, a successful login request will return res.status(200).json({ msg: "successfully logged in", status: 200}); , therefore we need an interface that defines this. Our interface might look like:

export interface LoginResponse {
  msg: string;
  status: number;
}

Which can be used in the service like so:

const login = async ({
  password,
  username
}: {
  password: string;
  username: string;
}): Promise<LoginResponse> => {
  const loginPayload = {
    password,
    username
  };

  return await AxiosService.POST<LoginResponse>(
    `${baseRoute}/login`,
    loginPayload
  );
};

Understanding AxiosResponse<T>

In the Axios library, AxiosResponse<T> is a generic interface that represents the structure of the response returned from an Axios HTTP request. It provides a strongly typed way to work with API responses, ensuring that your code can safely and predictably interact with the returned data.

The T in AxiosResponse<T> represents the expected data type of the response body. By specifying T, you inform TypeScript about the shape of the data you anticipate, enabling features like type checking and IntelliSense support.

Here’s a breakdown of the properties within the AxiosResponse<T> interface:

  • data: T
    The actual data returned from the API request, which matches the type T specified in the generic.
    Example: If the API returns a JSON object { name: "John", age: 30 }, and T is { name: string; age: number }, data will hold this object.

  • status: number
    The HTTP status code returned by the server (e.g., 200 for success, 404 for not found).

  • statusText: string
    A string representation of the HTTP status code (e.g., "OK" for 200, "Not Found" for 404).

  • headers: Record<string, string>
    The response headers returned by the server.

  • config: AxiosRequestConfig
    The configuration object that was used for the request.

  • request: any
    The raw request object associated with the response, useful for debugging.

Example of an AxiosResponseObject

{
  data: {
    id: 1,
    name: "John Doe",
    email: "john.doe@example.com"
  },
  status: 200,
  statusText: "OK",
  headers: {
    "content-type": "application/json",
  },
  config: {
    url: "/api/users/1",
    method: "get",
    headers: {
      Accept: "application/json, text/plain, */*"
    },
    baseURL: "https://example.com",
    timeout: 60000,
    withCredentials: false
  },
  request: {
    // The raw HTTP request object (implementation-specific, varies by environment)
  }
}

Understanding AxiosRequestConfig

AxiosRequestConfig is an interface provided by the Axios library to define the configuration options for an HTTP request. It allows you to customize the behavior of a request, including URL parameters, headers, serialization, and other HTTP settings. This flexibility is crucial for handling various use cases, such as passing query parameters, defining timeouts, or specifying custom serializers.

Key Properties of AxiosRequestConfig

Here are some commonly used properties in AxiosRequestConfig:

  • headers: Record<string, string>
    Custom headers to send with the request.

  • params: Record<string, any>
    Query parameters to include in the URL.

  • paramsSerializer: (params: any) => string
    A function to customize how query parameters are serialized into the URL.

Example Usage of AxiosRequestConfig

The following example demonstrates how AxiosRequestConfig is used in a GET request to pass query parameters and customize serialization:

const byId = async ({ id }: { id: string }): Promise<Asset> => {
  const queryParams: AxiosRequestConfig = {
    params: {
      _id: id // Query parameter passed to the request
    },

    /**
     * Serializes the query parameters for the request.
     * Ensures proper handling of non-string values and null checks.
     * @param params The query parameters to serialize.
     */
    paramsSerializer: params => {
      const searchParams = new URLSearchParams();
      Object.entries(params).forEach(([key, value]) => {
        if (value !== null) searchParams.append(key, String(value));
      });
      return searchParams.toString();
    }
  };

  return await AxiosService.GET<Asset>(`${baseRoute}/asset`, queryParams);
};

POST

This example demonstrates how to create a new resource using createResource and handle the result or any potential errors.

const handleCreateResource = async () => {

  const result = await ResourceService.createResource({
    name: "New Resource",
    description: "Description of the new resource"
  });

  if(result){
    console.log("Success")
  }
};

In this example, createResource is called with data matching the Resource structure, but without id and success (assuming they are managed by the server).

const handleCreateResource = async () => {

  const newResourceData: Omit<Resource, "id" | "success"> = {
    name: "New Resource",
    description: "Description of the new resource"
  };

  const result = await ResourceService.createResource(newResourceData);

  if(result){
    console.log("Success")
  }
};

PUT

This example shows how to update an existing resource using updateResource with an id and data, then handle the result.

const handleUpdateResource = async (id: string) => {

  const result = await ResourceService.updateResource(id, {
    name: "Updated Name",
    description: "Updated Description"
  });

  if(result){
    console.log("Success")
  }
};

In this example, updateResource is called with an id and updated data according to the Resource structure, using Omit to exclude id and success.

const handleUpdateResource = async (id: string) => {

  const updatedResourceData: Omit<Resource, "id" | "success"> = {
    name: "Updated Resource Name",
    description: "Updated description for the resource"
  };

  const result = await ResourceService.updateResource(id, updatedResourceData);

  if(result){
    console.log("Success")
  }
};
